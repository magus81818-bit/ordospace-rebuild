# CODE GUIDE 06 — 품질을 지키는 법

> 코드를 고칠 때마다 "내가 망가뜨린 건 아닌지?"를 확인해주는 자동 검증 시스템 해설.
> 검증 스크립트는 **코드를 실행해보는 게 아니라, 코드의 규칙을 점검**하는 도구입니다.

---

## 1. 전체 그림 — 품질을 지키는 4겹 방어막

```text
┌─────────────────────────────────────────────────────────────────┐
│  1층: static:validate-lifecycle    (라이프사이클 규칙 검증)       │
│       → 카드 상태 전이가 설계대로 작동하는가?                      │
├─────────────────────────────────────────────────────────────────┤
│  2층: static:validate-components   (부품 공장 검증)              │
│       → XSS 이스케이프, 마커 클래스, 35종 존재 확인              │
├─────────────────────────────────────────────────────────────────┤
│  3층: runtime-qa.js                (브라우저 런타임 QA)           │
│       → 역할별 화면이 실제로 뜨고, 내용이 채워지는가?             │
├─────────────────────────────────────────────────────────────────┤
│  4층: smoke-check.js               (헤드리스 스모크 테스트)       │
│       → Chrome을 자동으로 열어 모든 라우트가 살아있는지 훑기      │
└─────────────────────────────────────────────────────────────────┘
```

**실행 순서 권장**: 1층 → 2층 → 3층/4층 (빠른 것부터 느린 것 순)

## 2. 1층 — validate-static-module-lifecycle

| 항목 | 내용 |
|---|---|
| **파일** | `tools/validate-static-module-lifecycle.cjs` |
| **실행** | `npm run static:validate-lifecycle` |
| **소요** | 약 1초 |
| **브라우저 필요** | 아니요 (Node.js VM) |

### 무엇을 검증하는가

카드의 상태 전이 규칙(lifecycle 서비스)이 올바른지 **6가지 시나리오**를 재현합니다:

1. **QC 미완료 차단** — 체크리스트를 전부 체크하지 않으면 리뷰 요청 불가
2. **QC 완료 → 리뷰** — 체크리스트 완료 + 작업 기록 추가 후 submit → `review` 상태
3. **admin → client 전달** — PM이 확인 후 전송 → `done` + 클라이언트 알림
4. **client 수정 요청** — 클라이언트가 수정 요청 → `revision` + revisionCount 증가
5. **client 승인** — 승인 → `approved` 상태
6. **localStorage hydrate** — 저장 후 새 VM에서 로드해도 상태가 유지되는가

### 작동 원리 (비개발자용)

```text
[Node.js VM = "가짜 브라우저"] 안에서
  ├─ workspace.data.js (카드 12장) 로드
  ├─ lifecycle 서비스 로드
  ├─ 시나리오 1~5 순서대로 실행, assert로 확인
  └─ persist → 새 VM에서 hydrate → 시나리오 6 확인
```

"가짜 브라우저"라서 실제 화면은 안 뜨지만, 상태 전이 규칙은 진짜와 동일하게 테스트됩니다.

### 실패하면?

- lifecycle 서비스의 가드 조건을 변경했는데 이 테스트를 안 고쳤을 때
- 새 상태를 추가했는데 테스트에 반영하지 않았을 때

→ 에러 메시지가 "어떤 assert가 실패했는지" 알려줍니다.

## 3. 2층 — validate-static-components

| 항목 | 내용 |
|---|---|
| **파일** | `tools/validate-static-components.cjs` |
| **실행** | `npm run static:validate-components` |
| **소요** | 약 1초 |
| **브라우저 필요** | 아니요 (Node.js VM) |

### 무엇을 검증하는가

부품 공장(components)의 **35개 팩토리**가 안전하고 정확한지 4가지를 확인합니다:

1. **네임스페이스 완전성** — `ORDO_UI_COMPONENTS`에 35종이 모두 등록됨
2. **XSS 이스케이프** — 19개 팩토리에 `<script>alert(1)</script>` 를 넣어보고, 출력에 `<script>`가 그대로 남아있으면 실패
3. **마커 클래스 보존** — `ordo-c-status-badge`, `ordo-c-kpi-card` 같은 필수 CSS 클래스가 출력에 있는지
4. **화면의 규칙 준수** — workspace screen이 ModuleCard를 직접 만들거나 card.status를 직접 바꾸지 않는지

### XSS 검증의 의미

```text
입력: '<script>alert(1)</script>"&'
     ↓ 팩토리 호출
기대 출력: '&lt;script&gt;alert(1)&lt;/script&gt;&quot;&amp;'  ← 무해한 텍스트
실패 출력: '<script>alert(1)</script>'  ← 코드가 실행될 수 있는 위험!
```

→ 이 검증 덕분에 "사용자가 제목에 코드를 입력하면 보안 문제가 생기는" 사고를 미리 잡습니다.

### 마커 클래스란?

특정 팩토리에 반드시 있어야 하는 CSS 클래스입니다. 예:
- `StatusBadge` → `ordo-c-status-badge`
- `MetricCard` → `ordo-c-kpi-card`
- `ModuleCardListItem` → `ordo-c-module-card`, `bar-l`

이 클래스들은 스타일링과 QA 선택자에 사용되므로, 실수로 지우면 화면이 깨집니다.

## 4. 3층 — runtime-qa.js (브라우저 런타임 QA)

| 항목 | 내용 |
|---|---|
| **파일** | `app/qa/runtime-qa.js` |
| **실행** | 브라우저 콘솔에서 수동, 또는 개발 모드에서 자동 |
| **소요** | 약 2~5초 |
| **브라우저 필요** | 예 |

### 무엇을 검증하는가

**ROUTE_CASES 배열**에 정의된 각 라우트를 순회하면서:

1. 해당 역할(role)로 세션을 설정
2. 해시를 이동 (예: `#admin-cards`)
3. 화면 DOM 요소(`screenId`)가 존재하는지 확인
4. 렌더 타겟(목록, KPI 등)에 내용이 채워졌는지 확인

### 1·2층과의 차이

| 구분 | 1·2층 (static) | 3층 (runtime) |
|---|---|---|
| 실행 환경 | Node.js VM (가짜) | 실제 브라우저 |
| 검증 대상 | 규칙·팩토리 로직 | 화면이 실제로 그려지는가 |
| 속도 | 1초 | 2~5초 |
| 발견하는 문제 | 규칙 위반, XSS, 클래스 누락 | 라우팅 오류, 렌더 실패 |

## 5. 4층 — smoke-check.js (헤드리스 스모크)

| 항목 | 내용 |
|---|---|
| **파일** | `app/qa/smoke-check.js` |
| **실행** | `npm run smoke` |
| **소요** | 약 10~20초 |
| **브라우저 필요** | Chrome이 설치되어 있어야 함 (자동 실행) |

### 무엇을 검증하는가

Chrome을 **headless(화면 없이)** 모드로 띄워서:

1. 각 라우트로 이동
2. DOM 쿼리로 화면 요소 존재 확인
3. 내용이 비어있지 않은지 확인

### 작동 원리 (비개발자용)

```text
[Node.js 스크립트] 
  → Chrome 실행 (화면 안 띄움)
  → Chrome DevTools Protocol(CDP)로 "이 주소로 가" 명령 전송
  → "이 요소가 있나?" 질문 → 있으면 OK, 없으면 FAIL
  → Chrome 종료
```

"로봇이 앱을 열어서 모든 페이지를 눌러보는 것"과 같습니다.

### 3층과의 차이

- 3층: 이미 열려있는 브라우저에서 검사 (개발 중 빠른 확인)
- 4층: Chrome을 새로 띄워서 처음부터 검사 (배포 전 최종 확인)

## 6. 언제 어떤 검증을 돌려야 하는가

| 상황 | 돌릴 검증 | 이유 |
|---|---|---|
| lifecycle 함수를 고쳤다 | 1층 | 상태 전이 규칙이 깨지지 않았나 |
| components/ 팩토리를 수정했다 | 2층 | XSS·클래스 유지 확인 |
| 라우트나 화면을 추가/삭제했다 | 3·4층 | 화면이 실제로 뜨는지 |
| 뭘 고쳤는지 잘 모르겠다 | 전부 | 1분이면 끝남 |
| 배포 직전 | 전부 (특히 4층) | 최종 안전망 |

**한 줄 요약**: 의심되면 `npm run check:js && npm run static:validate-lifecycle && npm run static:validate-components`를 돌리세요. 1분이면 끝납니다.

## 8. 자주 묻는 질문 (FAQ)

**Q: "static"이라는 이름인데 정적 분석인가요?**
A: 반쯤 맞습니다. 코드를 읽기만 하는 게 아니라 VM에서 실제로 실행해봅니다. 하지만 브라우저 없이 돌아가므로 "정적(static)"이라 부릅니다. "반실행 검증"이 더 정확한 표현.

**Q: 테스트가 실패하면 앱이 망가진 건가요?**
A: 앱 자체는 멀쩡할 수 있습니다. 하지만 "규칙 위반"이 발생한 것이므로, 배포 전에 반드시 고쳐야 합니다. 예: XSS 테스트 실패 = 지금 당장 보안 문제는 아니지만, 악의적 데이터가 들어오면 위험해질 수 있다는 경고.

**Q: smoke-check가 Chrome을 못 찾는다고 합니다.**
A: Chrome이 설치되어 있어야 합니다. 경로를 자동으로 찾지만, 특이한 위치에 설치했다면 스크립트의 Chrome 경로 탐색 로직을 확인하세요.

**Q: 팩토리를 새로 추가했는데 테스트가 실패해요.**
A: `validate-static-components.cjs`의 `expected` 배열에 새 팩토리 이름을 추가하고, XSS 케이스도 추가해주세요. "공장에 새 기계를 들였으면 검사 목록에도 추가"하는 것.

**Q: 이 테스트들은 얼마나 자주 돌려야 하나요?**
A: 코드를 한 글자라도 고쳤으면 돌리세요. 걸리는 시간은 4개 합쳐 30초 미만입니다.

## 9. 5분 따라하기 — 검증 스크립트 체험

1. 터미널을 열고 프로젝트 루트로 이동합니다.
2. 1층 실행:
   ```bash
   npm run static:validate-lifecycle
   ```
   → `{"ok": true, "checked": [...]}` 이 나오면 성공.

3. 2층 실행:
   ```bash
   npm run static:validate-components
   ```
   → `{"ok": true, "checked": [...]}` 이 나오면 성공.

4. (심화) `tools/validate-static-components.cjs`를 열어서 `expected` 배열을 보세요.
   → 35개 이름이 곧 부품 공장의 "출석부"입니다.

5. (실험) 일부러 `base.ui.js`의 `escapeHtml` 함수에서 `<` 치환을 주석 처리하고 2층을 돌려보세요.
   → XSS 테스트가 실패하는 것을 확인한 뒤 **반드시 원래대로 되돌리세요**.
