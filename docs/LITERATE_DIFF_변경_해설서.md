# LITERATE DIFF — 변경 해설서

> 이 문서는 Geoffrey Litt의 "literate diff" 형식을 따릅니다.
> 코드가 **왜, 어떤 순서로, 어떤 판단 아래** 바뀌었는지를 서사로 설명합니다.
> diff를 알파벳순 파일 목록으로 나열하지 않고, **이야기 순서**로 풀어씁니다.

---

## 0. 이 문서를 읽기 전에 알아야 할 것

ORDOSPACE는 **역할 3개(admin/worker/client)의 협업 워크스페이스**입니다.
카드(ModuleCard)라는 작업 단위가 `pending → in_progress → review → done → approved`를 거치며,
브라우저(localStorage)와 서버(Postgres)에 이중 저장됩니다.

이 앱은 React 같은 프레임워크 없이, **전역 `<script>` 태그 + HTML 문자열**로 만들어져 있습니다.
번들러도 없습니다. `index.html`이 30개 넘는 JS 파일을 순서대로 로드합니다.

**이 문서가 다루는 두 단계:**
- **Phase A** (컴포넌트 분리 리팩터) — 12라운드에 걸쳐 중복 코드를 정리
- **Phase C** (코드 주석화 + 가이드) — 10라운드에 걸쳐 비개발자 해설 추가

---

## 1. 바꾸기 전: 무엇이 문제였나

### 1-1. 같은 코드가 4벌씩 복사되어 있었다

카드를 화면에 그리는 코드가 admin, worker, client 화면 파일에 **각각 따로** 있었습니다.

```
admin-workspace.screen.js    → adminCardListItemHtml()    (카드 목록)
worker-workspace.screen.js   → workerCardSummaryHtml()    (카드 목록)
client-workspace.screen.js   → projectModuleCard()        (카드 목록)
workspace.ui.js              → moduleCard()               (공용 카드)
```

4개 함수가 거의 같은 HTML을 만들지만 미묘하게 달랐습니다.
상태 배지, QC 체크리스트, 코멘트 목록, 첨부파일 목록도 마찬가지 — 전부 3~4벌씩.

**한 곳을 고치면 나머지 3곳도 같이 고쳐야** 하는데, 잊어버리면 화면마다 다르게 보입니다.

### 1-2. 공유 유틸이 엉뚱한 파일에 살고 있었다

`statusBadgeHtml`(상태 배지)는 `client-workspace.screen.js`에 정의되어 있었는데,
admin과 worker 화면도 이 함수를 썼습니다.

이게 돌아가는 이유는 `index.html`의 스크립트 로드 순서 때문이었습니다:
client 파일이 먼저 로드되니까 전역에 함수가 등록되고, 나중에 로드되는 admin/worker가 쓸 수 있었던 것.

**누군가 client 파일의 로드 순서를 바꾸면 admin/worker가 깨지는** 시한폭탄이었습니다.

### 1-3. 죽은 코드가 파일의 상당 부분을 차지했다

리팩터 과정에서 lifecycle 서비스가 도입되면서, 직접 상태를 바꾸는 구버전 함수가 남아있었습니다.
같은 이름의 함수를 아래에서 **재할당**(덮어쓰기)하는 패턴:

```js
function bindAdminCardActions() { /* 구버전 — 실행 안 됨 */ }
// ... 200줄 뒤 ...
bindAdminCardActions = function() { /* 신버전 — 이것만 실행 */ }
```

읽는 사람은 구버전을 읽고 "아, 이렇게 동작하는구나" 하고 넘어가다가,
한참 뒤에야 재할당을 발견하고 혼란에 빠집니다.

### 1-4. 코드에 해설이 전혀 없었다

41개 JS 파일 중 파일 헤더(이 파일이 무엇인지 알려주는 주석)가 있는 파일이 **0개**였습니다.
비개발자가 코드를 열면 무엇을 보고 있는지조차 알 수 없었습니다.

## 2. Phase A — 컴포넌트 분리 리팩터 (12라운드)

> 핵심 판단: "4벌의 중복을 1벌로 모으되, 화면의 픽셀 하나도 바꾸지 않는다."

### 2-1. 전략: 왜 이 순서로 했는가

리팩터의 순서는 **의존성 방향**을 따랐습니다:

```
유틸(base) → 상태(status) → 메트릭(metric) → 카드(module-card) → 상세(detail) → 폼(form)
```

아래쪽(유틸)부터 만들고, 위쪽(화면)은 나중에 교체합니다.
이유: 위쪽을 먼저 바꾸면 아래쪽이 없어서 깨지지만, 아래쪽을 먼저 만들면 위쪽은 아직 구코드를 쓰고 있어도 됩니다.

### 2-2. 라운드 1 — 베이스라인 고정 (코드 무변경)

아무것도 바꾸지 않고, **"지금 이 상태가 정상"이라는 기록**만 남겼습니다.

```
npm run check:js                    → 통과
npm run static:validate-lifecycle   → 통과
npm run smoke                       → 통과
```

이 기록이 있어야 나중에 "내가 망가뜨린 건가, 원래 이랬던 건가?"를 판단할 수 있습니다.

### 2-3. 라운드 2~4 — 부품 공장의 기반 다지기

**라운드 2: `base.ui.js` 신설**

4벌씩 흩어져 있던 공용 유틸을 한 곳으로 모았습니다:

| 함수 | 원래 있던 곳 | 옮긴 곳 |
|---|---|---|
| `escapeHtml` | 각 파일에 `moduleEsc`, `roomEsc` 등으로 중복 | `base.ui.js` (1벌) |
| `safePct`, `todayDateText` 등 | `client-workspace.screen.js` | `base.ui.js` |
| `routeParams` | `worker-workspace.screen.js` | `base.ui.js` |

원래 파일의 함수 이름은 **위임 래퍼**로 유지했습니다:
```js
// workspace.ui.js — 기존 이름 보존, 본체는 base로 위임
function moduleEsc(v) { return window.ORDO_UI_COMPONENTS.escapeHtml(v); }
```

이렇게 하면 기존 호출부를 한 줄도 안 고쳐도 됩니다. "로드 순서 시한폭탄"이 해소됐습니다.

**라운드 3: `status.ui.js` — tone 시스템 통합**

카드의 "기분"을 색으로 표현하는 tone 시스템(crit/warn/pend/ok/rej)이 여기저기 흩어져 있었습니다.
이것을 `status.ui.js` 한 파일로 모으고, `StatusBadge` 팩토리를 만들었습니다.

**라운드 4: `metric.ui.js` — KPI 카드 통합**

`MetricCard`, `ProgressTrack`, `EmptyState`, `Notice` 팩토리를 만들어
admin/worker/client에 흩어진 메트릭 카드와 빈 상태 패널을 흡수했습니다.

### 2-4. 라운드 5~6 — 가장 큰 절개

**라운드 5: `ModuleCardListItem` — 4벌의 카드를 1벌로**

이 프로젝트에서 가장 큰 변경이었습니다. 4개 함수가 만들던 카드 HTML을 하나의 팩토리로 통합했습니다.

4벌의 차이는 사실 3가지뿐이었습니다:
1. HTML 태그 (`<button>` vs `<article>`)
2. data 속성 이름 (`data-admin-card-id` vs `data-worker-card-id`)
3. 메타라인 구성 (admin은 게이트 번호 표시, worker는 체인 표시)

이 차이를 **옵션 객체(`opts`)**로 흡수:
```js
ModuleCardListItem(card, {
  dataAttr: 'data-admin-card-id',   // 화면마다 다른 부분
  metaParts: [card.spec, card.gateRef],
  selected: card.id === selectedId
})
```

**라운드 6: `detail.ui.js` — 상세 패널 9종 통합**

카드를 클릭하면 나오는 상세 패널도 3벌이었습니다.
이것을 9개의 작은 프리미티브로 분해:

```
DetailHeader → MetaGrid → QcList → WorkLogList →
AttachmentList → CommentList → DetailSection → ActionToolbar
```

각 프리미티브는 독립적이라, 화면마다 필요한 조합만 골라 쓸 수 있습니다.

### 2-5. 라운드 7~10 — 화면별 레거시 제거

이제 팩토리가 준비되었으니, 각 화면 파일에서 구버전 코드를 삭제했습니다.

| 라운드 | 화면 | 삭제한 것 |
|---|---|---|
| 7 | admin | 죽은 구버전 함수들, 전 화면 팬아웃(모든 화면을 다시 그리던 비효율) |
| 8 | worker | `workerCardSummaryHtml`, `workerDetailHtml`, 구 바인딩 |
| 9 | client project | `projectModuleCard`, `cardDetailBodyHtml`, 구 모달 |
| 10 | client approvals | `approvalDetailHtml`, **문자열 치환 몽키패치** 제거 |

**라운드 10의 몽키패치 제거**가 특히 중요했습니다.
client approvals 화면에는 이런 코드가 있었습니다:

```js
// 구코드 (몽키패치) — 버튼을 문자열 치환으로 추가
html = html.replace('</div><!-- toolbar-end -->',
  '<button data-approve>승인</button></div><!-- toolbar-end -->');
```

이것을 `ActionToolbar`의 버튼 배열에 정식으로 포함시켜 제거했습니다.

### 2-6. 라운드 11~12 — 마무리

**라운드 11: Form/Sheet 프리미티브**
일괄 생성(bulk create) 화면의 폼과 시트 오버레이를 팩토리로 교체.

**라운드 12: QA 상설화 + 최종 검증**
`validate-static-components.cjs` 검증 스크립트를 신설.
35개 팩토리의 XSS 이스케이프와 마커 클래스를 자동 검증.

## 3. Phase C — 코드 주석화 + 가이드 맵 (10라운드)

> 핵심 판단: "코드에 한 줄도 로직을 추가하지 않고, 주석과 빈 줄만 추가한다."

### 3-1. 왜 주석화가 필요했나

리팩터로 코드 구조는 깨끗해졌지만, **비개발자가 읽을 수 있는 상태는 아니었습니다.**

- 41개 JS 파일 중 "이 파일이 무엇인지" 알려주는 헤더가 0개
- 함수 위에 "이 함수가 무엇을 하는지" 설명이 없음
- 전체 구조를 그림으로 보여주는 문서가 없음

### 3-2. 주석화 규칙

모든 주석은 다음 규칙을 지켰습니다:

1. **코드 실행에 영향을 주는 변경 절대 금지** — 주석과 빈 줄만 추가/수정
2. **한국어, 비개발자 톤** — 가끔 비유 사용 ("우체국 창구", "금고지기")
3. **파일 헤더 6항목**: `[이 파일은]`, `[언제 실행]`, `[주요 등장인물]`, `[연결]`, `[다음 읽을 파일]`, `[수정할 때 주의]`
4. **주석 밀도 30% 이하** — 코드보다 주석이 많으면 안 됨
5. **매 라운드 종료 시 검증 3종 통과**

### 3-3. 라운드 R1 — 뼈대 구축

문서 폴더(docs/)를 만들고, 전체 구조를 보여주는 `CODE_GUIDE_00_시작하기.md`를 작성했습니다.
샘플 주석을 1개 파일에 넣어 톤과 형식을 검수받았습니다.

### 3-4. 라운드 R2 — 부팅 스토리

"앱을 열면 무슨 일이 일어나는가?"를 설명하는 가이드 챕터 01을 작성했습니다.

대상 파일 6개:
```
index.html → app.config.js → session.service.js →
app-shell.js → hash-router.js → module-card-lifecycle.service.js
```

이 순서가 바로 앱이 켜질 때 코드가 실행되는 순서입니다.
가이드도 이 순서를 따라 **"부팅 스토리"**로 풀어썼습니다.

### 3-5. 라운드 R3~R5 — 데이터와 부품 해설

| 라운드 | 가이드 | 설명 대상 |
|---|---|---|
| R3 | 02 데이터 사전 | 카드 한 장의 구조 (22개 필드), Room 데이터, 쿼리 헬퍼 |
| R4 | 03 컴포넌트 사전 전반 | base/status/metric 팩토리 — tone 시스템, XSS 방어 |
| R5 | 03 컴포넌트 사전 후반 | card/detail/form/sheet 팩토리 — 옵션 흡수 패턴 |

### 3-6. 라운드 R6 — 카드의 일생 (핵심 챕터)

가이드 04는 이 프로젝트의 **핵심 비즈니스 로직**을 설명합니다:

```
pending → in_progress → review → done → approved
                                   ↓
                               revision (수정요청)
```

각 전이에서 **누가 버튼을 누르고, 어떤 가드가 확인하고, 무엇이 저장되는지**를
비개발자도 따라갈 수 있는 톤으로 풀어썼습니다.

### 3-7. 라운드 R7~R8 — 화면과 서버

**R7**: 6개 screen 파일에 파일 헤더 + 섹션 배너 추가.

각 섹션 앞에 배너를 넣어 "지금 어디를 읽고 있는지" 알 수 있게 했습니다.

**R8**: API 핸들러(우체국 창구)와 repository(금고지기)에 주석 추가.

### 3-8. 라운드 R9~R10 — 품질과 마무리

**R9**: 검증/QA 스크립트 4개에 주석 추가 + 가이드 06 "품질을 지키는 법" 작성.

**R10**: 나머지 12개 미주석 파일에 헤더 추가 + 가이드 07 "총정리" 작성.
최종 결과: **41개 파일 전체에 파일 헤더 완료**.

## 4. 두 Phase를 관통하는 판단 원칙

### 4-1. "하나의 라운드 = 하나의 절개"

매 라운드는 **딱 한 가지**만 바꾸고, 반드시 검증을 통과한 뒤 다음으로 넘어갔습니다.

이유: 한 번에 여러 곳을 바꾸면, 깨졌을 때 "어디서 깨졌는지" 찾기 어렵습니다.
한 라운드에서 깨지면 그 라운드의 변경만 되돌리면 됩니다.

### 4-2. "신코드를 먼저 만들고, 구코드는 검증 후에만 삭제"

구코드를 먼저 지우면, 신코드가 문제가 있을 때 돌아갈 곳이 없습니다.
반드시 신코드가 동작하는 것을 확인한 뒤에야 구코드를 삭제했습니다.

### 4-3. "위임 래퍼로 기존 이름 보존"

`moduleEsc` → `escapeHtml`로 바꿀 때, `moduleEsc`를 삭제하지 않고
`escapeHtml`을 호출하는 1줄 래퍼로 남겼습니다.

이유: 기존 코드에서 `moduleEsc`를 호출하는 곳이 수십 군데인데,
전부 한꺼번에 바꾸면 실수 위험이 높습니다.
래퍼를 남기면 **호출부 수정 0건**으로 안전하게 전환됩니다.

### 4-4. "코드 실행에 영향을 주는 주석화 변경 절대 금지"

Phase C에서는 주석과 빈 줄만 추가했습니다.
변수명 변경, 로직 수정, 포맷팅 변경 — 전부 금지.

이유: 주석을 추가하다가 실수로 코드를 건드리면,
"주석 때문에 앱이 깨졌다"는 신뢰 파괴가 일어납니다.
주석화와 코드 변경은 반드시 별도 라운드로 분리합니다.
