## 1. 이 작업의 목표 (왜 하는가)

> "직접 읽고 관리할 수 있는 코드가 아니면 그 코드는 괴물이 되기 쉽다. 감당 가능한 범위까지만 손대라." — 강사 조언

이 프로토타입은 오너가 생각하는 **기능의 마지노선**이므로 범위를 줄일 수는 없다. 그렇다면 남은 방법은 하나 — **감당 가능한 범위를 코드 쪽으로 넓히는 것**이다. 이 작업이 끝나면:

1. 오너가 아무 파일이나 열어도 **첫 화면(파일 헤더)에서 "이 파일이 뭐 하는 물건인지"** 알 수 있다.
2. `docs/`의 가이드 챕터를 순서대로 읽으면 **"클릭 → 코드 → 저장 → 화면 갱신"의 전체 그림**이 머릿속에 그려진다.
3. 각 파일 헤더의 **[다음 읽을 파일] 이정표**를 따라가면 문서 없이 코드만으로도 투어가 이어진다.
4. 각 챕터의 **"5분 따라하기"**로 읽은 내용을 브라우저에서 직접 눈으로 확인한다 — 읽기만 한 지식은 증발하지만, 손으로 확인한 지식은 남는다.

성공 기준: **"이 버튼을 누르면 무슨 일이 일어나서 어디에 저장되는가?"라는 질문에 오너가 코드를 열어 스스로 답을 찾을 수 있다.**

## 2. 결과물 3종

### A. 코드 내 주석 (전 파일)

모든 JS 파일에 4단계 주석을 단다. **한국어**, 비개발자 기준, 필요하면 비유 1개까지 허용.

**① 파일 헤더 (모든 파일 공통 템플릿 — 이정표 포함):**

```js
/* ============================================================
   [이 파일은] ModuleCard의 상태 변경 규칙과 저장을 담당하는 "심판"
   [언제 실행] 페이지가 열릴 때 한 번 로드되고, 이후 버튼을 누를 때마다 호출됨
   [주요 등장인물]
     - submitWorkerReview : 작업자가 "리뷰 요청" 버튼을 눌렀을 때
     - approveClient      : 클라이언트가 "승인" 버튼을 눌렀을 때
     - persist            : 바뀐 내용을 브라우저와 서버에 저장
   [연결] ← 세 워크스페이스 화면이 호출 / → localStorage와 /api/module-cards에 저장
   [다음 읽을 파일] app/screens/worker-workspace.screen.js (이 규칙을 쓰는 화면)
   [수정할 때 주의] 상태 이름(pending, review...)을 바꾸면 서버 검증도 같이 바꿔야 함
   ============================================================ */
```

**② 섹션 구분 주석:** 파일 안의 큰 덩어리마다 `/* ── 필터 상태 관리: 사용자가 고른 필터를 기억한다 ── */`

**③ 함수 주석:** 목적 / 입력 / 반환 / 부수효과(화면 갱신·저장·알림 발송 여부). 부수효과가 특히 중요 — "이 함수를 부르면 저장까지 된다"를 모르면 코드를 그릴 수 없다.

**④ 줄 주석:** "왜 이렇게 했는지"가 자명하지 않은 줄에만. (예: `// 400ms 기다렸다 저장 — 연타로 서버를 두드리지 않기 위한 지연`)

**분량 규칙 (과밀 방지):**
- 주석이 파일 줄 수의 30%를 넘지 않게 한다 — 주석이 너무 많으면 그것도 읽기 장벽이다.
- 자명한 코드에는 달지 않는다 (`count + 1 // 1을 더한다` 금지).
- 기존 한국어 주석은 유지하고, 틀렸거나 낡은 것만 교정한다.
- 용어가 처음 등장하는 곳에서만 풀어 쓴다 (예: "hydrate(저장된 데이터를 불러와 되살리기)") — 이후엔 용어집 참조.

**index.html 특별 규칙:** 237KB 마크업 전체에 주석을 달지 않는다. 대신 최상단에 **"화면 지도" 주석 블록**(어느 줄 근처에 어느 화면 섹션이 있는지, 스크립트 로드 순서가 왜 이 순서인지)만 단다. 화면별 구분 주석은 이미 존재하므로 유지.

**가이드 작성 원칙:**
- 파일 구조 순서가 아니라 **유저 플로우(스토리) 순서**로 쓴다 — "작업자가 리뷰 요청을 누르면"으로 시작해 코드를 따라간다.
- 코드 참조는 라인 번호가 아니라 **함수명 + 파일명**으로 한다 (라인 번호는 코드가 바뀌면 거짓말이 된다).
- 각 챕터 끝에 **"5분 따라하기"** 실습 1~2개 (아래 C).
- 각 챕터 끝에 **"이 챕터를 읽고 답할 수 있어야 하는 질문 3개"** — 스스로 이해도를 점검하는 장치.

### C. "5분 따라하기" 체크포인트 (각 챕터 말미)

읽은 것을 브라우저에서 직접 확인하는 초단기 실습. 예시:

- (02 데이터) F12 콘솔에서 `window.ORDO_MODULE_CARDS[0]` 을 쳐서 카드 한 장의 실물을 본다.
- (03 컴포넌트) 콘솔에서 `ORDO_UI_COMPONENTS.StatusBadge('테스트', 'ok')` 를 실행해 "함수가 HTML 문자열을 만든다"를 눈으로 확인한다.
- (04 일생) worker로 로그인해 QC를 다 체크하고 리뷰 요청 → admin으로 전환해 리뷰요청 필터에서 그 카드를 찾는다.
- (05 저장) F12 → Application → Local Storage에서 `ordospace.static.moduleCards.v1` 키를 열어 방금 승인한 카드의 status가 `approved`로 저장된 것을 확인 → 새로고침 → 유지 확인.
- (06 품질) 터미널에서 `npm run static:validate-components` 를 직접 돌려 초록 결과를 본다.

## 4. 매 라운드 공통 프로세스 (5단계)

1. **주석 달기** — 헤더 → 섹션 → 함수 → 줄 순서로. 대상 파일의 기존 주석 검토·교정 포함.
2. **가이드 챕터 작성** — 해당 라운드 코드와 상호 링크(함수명 기준). 코드 이정표([다음 읽을 파일])도 이때 연결.
3. **따라하기 + 자가진단 질문 3개** 작성.
4. **검증 — 동작 불변 증명**:
   - `npm run check:js` (문법)
   - `npm run static:validate-lifecycle` + `static:validate-components` (동작·마크업)
   - `npm run smoke` (12 라우트 + Runtime QA 20/20)
   - `git diff` 육안 확인: **변경된 줄이 주석/공백뿐인지** (코드 줄 변경 0가 이 작업의 절대 규칙)
5. **피드백 루프 (교육 핵심)** — 사용자가 해당 챕터를 읽고 따라하기 실행 → 이해 안 되는 주석/문장을 알려줌 → 다음 라운드 첫머리에서 해당 부분을 고치고 시작. **"읽고 질문이 남는 주석은 실패한 주석"** 이 기준이다.

## 5. 안전 규칙

- **코드 실행에 영향을 주는 변경 절대 금지.** 주석과 빈 줄만 추가/수정한다. 리팩터링 욕심이 나는 지점은 고치지 말고 `[수정할 때 주의]` 또는 후속 항목 목록에 적는다.
- 이 작업 중 발견되는 버그/개선점은 코드에 손대지 않고 별도 목록(`docs/FINDINGS_DURING_ANNOTATION.md`)에 기록만 한다.
- Codex 제출 커밋 **이후** 시작 (확정) — 주석 라운드는 라운드당 1커밋으로 분리 가능해짐 ("docs: R2 부팅 스토리 주석" 식).
- react-mvp/, dist-react/, vendor/, tailwind.build.css(생성물), package-lock.json은 대상 제외.

## 6. 시작 조건과 첫 지시

- **시작 조건**: Codex의 제출 기준 커밋 완료 통보.
- **첫 라운드(R1)**: 사용자가 "R1 진행" 지시 → docs/ 뼈대 + 전체 지도 + lifecycle 서비스 샘플 주석 → **사용자가 샘플을 읽고 "이 톤/깊이 좋다 or 더 쉽게/더 간결하게" 피드백** → 확정 후 R2부터 본격 진행.
- 진행 중 언제든 "이 파일 먼저"라고 우선순위를 바꿔도 된다 — 라운드 순서는 교육적 권장 순서일 뿐 의존성이 강하지 않다.

## 7. 이 설계가 "이상적"이기만 한 게 아닌 이유

- 주석과 가이드는 코드와 함께 저장소에 들어가므로, 이후 어떤 AI 에이전트(Codex 포함)가 작업하든 **같은 맥락을 읽고 시작**한다 — 사람만을 위한 것이 아니라 협업하는 AI들의 온보딩 문서이기도 하다.
- 완료 기준이 명확하다: 오너가 "버튼 → 코드 → 저장" 질문에 스스로 답할 수 있으면 끝. 감당 가능한 범위가 실제로 넓어졌는지는 R10의 통독 시뮬레이션과 따라하기 체크포인트 통과로 측정한다.
